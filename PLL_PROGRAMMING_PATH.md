# MC145156 PLL programming path

This note records the current best match between the schematic/continuity
checks and the original `ST_M27256.HEX` firmware.

## Confirmed hardware signals

| MC145156P pin | Signal | Source |
| ---: | --- | --- |
| `11` | `clock` | 8031 pin `16`, `P3.6` / `/WR`, through `54LS09` |
| `12` | `data` | 8031 pin `13`, `P3.3`, through `54LS09` |
| `13` | `enable` | 8243 pin `5`, `P43`, through `54LS09` |

The 8243 that drives the PLL enable line is schematic `MN13`. Its 4-bit command
bus is driven from 8031 pins `5..8`, i.e. `P1.4..P1.7`. Its chip select is
driven directly by 8031 pin `1` / `P1.0`.

There is also a second 8243, schematic `MN14`; its chip select is driven from
the same `P1.0` line through an inverter. So `P1.0` is not just a periodic
watchdog-looking signal: it selects which 8243 receives the `P1.4..P1.7` +
`P3.5` transaction.

Additional RF/TX mapping on the same `MN13`:

- `MN13` pin `5` / `P43` controls PLL `EN`.
- `MN13` pin `4` controls microphone enable; it is probably the adjacent
  `P42` output, but the confirmed fact is the physical pin `4`.

Important point: on the 8031, `P3.6` is the external-memory write strobe
`/WR`. The firmware therefore does not use `SETB P3.6` or `CLR P3.6` for the
PLL clock. Clock pulses are generated by `MOVX @R0,A` external writes.

## Main PLL routine

The serial shift routine is at `L50B3` in `ST_M27256_annotated.asm`.

Evidence:

- `P3.3` is set/cleared before each bit.
- each `MOVX @R0,A` immediately after that generates one `/WR` pulse on
  `P3.6`, which is the PLL clock line.
- no direct `SETB/CLR P3.6` is needed or expected.

Excerpt:

```asm
L50B3:
50B3:  75 A0 00  MOV P2,#00h
50B6:  20 67 07  JB 2Ch.7,L50C0
50B9:  C2 B3     CLR P3.3       ; PLL data = 0
50BB:  F2        MOVX @R0,A     ; /WR pulse = PLL clock
50BC:  F2        MOVX @R0,A     ; second clock pulse for this path
50BD:  02 50 C6  LJMP L50C6
L50C0:
50C0:  D2 B3     SETB P3.3      ; PLL data = 1
50C2:  F2        MOVX @R0,A     ; /WR pulse = PLL clock
50C3:  C2 B3     CLR P3.3
50C5:  F2        MOVX @R0,A     ; /WR pulse = PLL clock
```

After that the routine shifts bits from `R4`, `R3`, and `R2`:

```asm
50C6:  EC        MOV A,R4
50C7:  03        RR A
...
50D1:  F2        MOVX @R0,A
...
50DD:  EB        MOV A,R3
...
5136:  EA        MOV A,R2
...
5184:  F2        MOVX @R0,A
```

So the likely firmware abstraction is:

1. compute divider/control fields into `R4:R3:R2`;
2. put data bit on `P3.3`;
3. perform `MOVX @R0,A` to pulse `/WR` / PLL clock;
4. repeat until the complete MC145156 word has been shifted;
5. toggle the external enable/load line through the 8243/latch path.

## Word calculation path

The PLL word is prepared by `L4FD9`.

Call path:

```text
main dispatcher L585F
  flag 25h.2
    -> L4FD9
       -> L5036 / L505C / L5089
          -> L50B3
```

`EVENT_DISPATCH.csv` marks `25h.2` as a one-shot event that jumps to `L4FD9`.

At `L4FD9`, the firmware reads the channel/frequency digits from internal RAM
locations selected by `2Ch.7`:

```asm
4FFD:  30 67 05  JNB 2Ch.7,L5005
5000:  78 1B     MOV R0,#1Bh
...
5005:  78 1D     MOV R0,#1Dh
5007:  E6        MOV A,@R0
5008:  12 51 98  LCALL L5198    ; BCD -> binary
500B:  75 F0 64  MOV B,#64h
500E:  A4        MUL AB
...
5013:  E6        MOV A,@R0
5014:  12 51 98  LCALL L5198    ; BCD -> binary
```

It then adds a correction/table value from the personalization/config EPROM
area:

```asm
501D:  90 04 65  MOV DPTR,#0465h
5020:  E0        MOVX A,@DPTR
5021:  C4        SWAP A
5022:  54 07     ANL A,#07h
5024:  23        RL A
5025:  90 04 A0  MOV DPTR,#04A0h
5028:  25 82     ADD A,DPL
502A:  F5 82     MOV DPL,A
502C:  E0        MOVX A,@DPTR
502D:  2C        ADD A,R4
...
5030:  E0        MOVX A,@DPTR
5031:  2B        ADD A,R3
```

The mode bits in `XDATA 0465h` choose one of three divider transformations:

| Branch | Condition | Final jump |
| --- | --- | --- |
| `L5036` | default, `ACC.6=0` and `ACC.5=0` | `5053 -> L50B3` |
| `L505C` | `ACC.6=1` | `5086 -> L50B3` |
| `L5089` | `ACC.5=1` | falls through to `L50B3` |

These branches divide/mix the value using constants `40h`, `28h`, or `14h`.
That is likely channel spacing or prescaler/reference mode handling.

## Enable/load line candidate

The MC145156 enable line is not driven directly by an 8031 bit. It comes from
8243 `P43` on `MN13`, so in firmware it is hidden behind the common 8243 write
routine `L03CC`:

```asm
L03CC:
03CC:  C2 B5     CLR P3.5
03CE:  F5 90     MOV P1,A
03D0:  D2 B5     SETB P3.5
03D2:  22        RET
```

With the MN13/MN14 chip-select clue, this routine should be interpreted as:

1. the caller first writes a select/command byte to `P1`;
2. `P1.0` selects which 8243 is active (`MN13` directly, `MN14` through an
   inverter);
3. `P1.4..P1.7` carry the 8243 command/data nibble;
4. `P3.5` is the 8243 `PROG`/strobe line;
5. the routine writes the second nibble/value on `P1`.

The low nibble in the observed `P1` values is therefore not arbitrary padding.
Values ending in `Eh` and `Fh` can select different 8243 chips because bit
`P1.0` changes.

The four writes tightly associated with PLL programming are:

| Address | P1 before call | A before call | Position |
| ---: | ---: | ---: | --- |
| `4FE0` | `CEh` | `7Eh` | before serial shift |
| `4FF4` | `8Eh` | `1Eh` | before serial shift |
| `518A` | `8Eh` | `8Eh` | after serial shift |
| `5192` | `CEh` | `7Eh` | after serial shift |

These are the best candidates for 8243 `P43` enable/load control. The exact
active polarity should be verified on `MN13` pin `5` and MC145156 pin `13` with
the scope while original firmware changes channel.

## Next decode step

To turn this into standalone C firmware, capture or emulate one channel change:

1. trigger on MC145156 pin `13` / enable;
2. capture MC145156 pins `11` clock and `12` data;
3. enter channels `0`, `1`, `2`, then `0` again;
4. compare the shifted bit stream with the `R4:R3:R2` values produced by
   `L4FD9`.

Once the bit order and enable polarity are confirmed, the C implementation can
use:

```c
PLL_DATA = bit;
dummy_movx_write(); /* pulse /WR on P3.6, the PLL clock */
```

and a small 8243 helper for the `P43` enable/load line.

## Relation to PTT/RX-TX state

PTT press/release handling does not directly shift the PLL word. It normally
sets RF state flags first, then the common dispatcher applies the current RF
state when one-shot flag `25h.2` reaches `L4FD9`.

Important links:

- `L468A` is the TX request/display marker. Its fallback path sets `25h.2` and
  `2Ch.7`, which causes `L4FD9` to use the `1Bh/1Ch` digit pair.
- `L4648` is the RX/return marker. It clears `2Ch.7`, sets `25h.2`, and queues
  panel operation `16h`.
- `L459D` and `L45B3` set `25h.6`, initialize RF timing bytes `60h/61h/4Eh`,
  and call `L5964`.
- `L5977` clears `20h.0` and `25h.6`, which is the compact TX/RF-disable
  primitive.

So for replacement firmware, keep PLL programming (`L4FD9/L50B3`) separate from
PTT state changes. PTT should first select TX or RX hardware state, then call a
PLL apply routine when the selected state or channel requires a new divider.
